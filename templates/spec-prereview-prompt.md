# Spec Pre-Review — Self-Exploration Mode

- Request: {{REQ_ID}} / Task: {{TASK_ID}}
- Spec: {{SPEC_PATH}}
- Escalation Mode: {{ESCALATION_MODE}}

## 목적

이 프롬프트는 구현 전 1st pass 질문 생성에만 사용된다. spec.md를 직접 읽고 모호하거나 누락된 사항을 질문 목록으로 출력하라.

**구현 금지**: 코드 수정, 파일 생성, git 명령 등 어떠한 구현 행위도 절대 금지한다. 질문만 생성하라.

## 자기탐색 지시

아래 순서로 스펙을 직접 탐색하라. PM이 제공한 요약에 의존하지 말고 원본 파일을 직접 읽어라.

1. 스펙 직접 읽기: `cat {{SPEC_PATH}}` (또는 Read 도구)
2. §2 변경 범위의 파일 목록을 파악하고, 해당 파일들을 실제로 열어 현재 상태를 확인하라
3. 다음 관점에서 이슈 목록을 생성하라:
   - **모호한 요구사항**: 두 가지 이상 해석이 가능한 표현
   - **누락된 엣지케이스**: spec에 언급 없는 예외 상황
   - **암묵적 가정**: spec이 사실로 전제하지만 코드베이스에서 확인이 필요한 것
   - **충돌 패턴**: 기존 코드베이스 방식과 spec의 접근법이 다른 경우
4. 각 이슈에 심각도를 부여하라:
   - **CRITICAL**: 구현 방향이 근본적으로 달라지는 모호함 또는 명확한 요구사항 누락 (반드시 사용자/PM 확인 필요)
   - **MAJOR**: 중요한 엣지케이스 누락, 암묵적 가정 중 검증 필요한 것, 기존 코드와 충돌 가능성 (PM 판단 필요)
   - **MINOR**: 사소한 명확화 필요, 선택적 개선 사항 (PM 자체 처리 가능)
5. 이슈 없으면 `NO_ISSUES` 한 줄 출력
6. 이슈 있으면 심각도별로 출력 (각 이슈에 spec §번호 또는 파일:라인 참조 포함)

## 에스컬레이션 모드 컨텍스트

`{{ESCALATION_MODE}}` 값에 따라 질문 작성 방식을 조정하라:

- **`user`** (plan → start 흐름, 사용자에게 에스컬레이션): 사용자가 직접 답변할 것이다. 배경 지식을 전제하지 말고 명확하고 구체적으로 이슈를 기술하라. 선택지가 있다면 함께 제시하라. CRITICAL/MAJOR 이슈를 우선 집중하라.
- **`pm-self`** (start-only 흐름, PM 자체 해소): PM이 코드베이스와 컨텍스트를 토대로 자체 판단할 것이다. 판단 근거가 될 수 있도록 관련 파일/패턴/트레이드오프를 이슈에 포함하라. MINOR 이슈는 PM이 자체 처리하므로 해결 방향을 함께 제안하라.

## 규칙

- 코드 수정/파일 생성/git 명령 절대 금지
- 질문만 생성하라 — 답변하거나 구현하려 하지 말 것
- spec §2의 변경 범위 파일을 직접 읽되, 수정은 금지
- 완료 전 모든 질문이 위 4가지 관점 중 하나에 해당하는지 self-check할 것

## 출력 형식

이슈가 없으면 첫 줄에 "NO_ISSUES"만 반환.

이슈가 있으면 아래 형식으로 반환:

```
CRITICAL: [spec §X 또는 파일:라인 참조] 이슈 내용 — (반드시 사용자/PM 확인 필요)
MAJOR: [spec §X 또는 파일:라인 참조] 이슈 내용 — (PM 판단 필요)
MINOR: [spec §X 또는 파일:라인 참조] 이슈 내용 — (PM 자체 처리 가능)
```

각 항목은 한 줄, 최대 10개.
